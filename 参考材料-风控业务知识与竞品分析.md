## 一、宝尊集团画像：你要服务的客户到底长什么样

### 1.1 业务版图

宝尊不是一家简单的电商公司，它是**品牌数字商业伙伴**，业务横跨三大板块：

| 板块 | 内容 | 2025年营收 | 关键特征 |
|------|------|-----------|----------|

**这意味着什么？** 宝尊的风控复杂度远超一般电商：

- **多实体结构**：港股+美股双上市（9991.HK / BZUN），需同时满足港交所、SEC、中国证监会的合规要求，内控审计压力大
- **2025年全年营收99亿，经营现金流4.2亿**，正处于三年转型收官、扭亏为盈的关键期——对成本管控和风控质量的要求只会更高

## 二、风控部的真实工作：不是教科书，是每天在干的事

### 2.1 风控部日常到底做什么

别用"风险识别、风险评估、风险应对"这种教科书语言去跟业务老大聊。他们每天实际做的事情是：

**日常高频工作**：
1. **审单**：每天有大量的采购申请、合同签署、付款申请需要风控审核
2. **查数**：在多个系统之间来回切换，手动核对同一笔业务在不同系统的记录是否一致
3. **追票据**：发票到了没？合同签了没？验收单有没有？三单/四单齐不齐？
4. **写报告**：定期向上汇报风控指标、异常事件、整改情况
5. **应付审计**：内审外审来了，要准备底稿、准备证据链

**真正的痛苦**：
- 不是"不知道有风险"，而是**信息散落在不同系统里，拼不起来**
- 一笔非经营性采购要看合同系统的合同、采购系统的采购单、财务系统的付款记录，三个系统三个界面三次登录三次查询
- 底下200人里做执行的人，痛苦是每天重复机械劳动
- 张文洁作为leader，痛苦是**看不到全貌**——他要结果、要数据，但数据要人工汇总，慢且容易遗漏

### 2.2 核心风控机制：三单匹配 / 四流一致

跟风控业务聊天，这两个术语必须烂熟于心：

**三单匹配**（采购订单 / 入库单 / 发票）：
```
采购订单 ←→ 入库验收单 ←→ 供应商发票
   ↑              ↑              ↑
  数量            数量            数量  → 差异≤2%
  单价            单价            单价  → 差异≤1%
  税率            税率            税率  → 必须100%一致
```
超阈值 → 冻结付款申请 → 人工复核 → 如果三个系统数据不在同一处，这个复核就是噩梦

**四流一致**（合同流 / 资金流 / 发票流 / 物流）：

这是金税四期下税务穿透式监管的"护身符"。四流中任何一流对不上，都可能被税务机关认定为虚开发票：

| 流 | 校验要点 |
|----|---------|
| **合同流** | 合同主体与实际交易方一致；品名、规格、数量、单价清晰；不能日期倒签 |
| **资金流** | 付款方、收款方必须与合同主体、发票抬头**完全一致**；网银备注需标注合同编号；严禁资金回流、私卡走账 |
| **发票流** | 品名型号数量与合同/出库单**严丝合缝**；发票代码+号码唯一性校验；验真（税局接口） |
| **物流** | 出入库单双方签字、物流合同、货运发票、司机轨迹、过磅单、签收回执 |

最终要形成**"四合一"底稿**：合同 + 银行回单 + 发票 + 物流单据，装订归档备查。

**对宝尊的特殊意义**：双上市公司，面临SEC和港交所的审计压力，四流一致不是"最好做到"，是"必须做到"。

### 2.3 付款审批的完整生命周期

这是风控部最核心的管控链条，每一步都是一个可能的 Agent 切入点：

```
① 发票验真与三单匹配
   ↓ 系统通过税局接口验真 → 匹配采购订单与收货记录
   ↓ 差异超阈值 → 冻结并复核

② 应付账款自动生成
   ↓ 匹配通过后生成应付行项目
   ↓ 包含：供应商、PO号、物料、税率、到期日、费用科目

③ 付款申请与分级审批
   ↓ 按金额/供应商等级/预算占用 → 分级审批
   ↓ 大额（如>50万）需CFO/法务联签

④ 支付执行与回单同步
   ↓ 银企直连批量付款 → 实时回传流水与电子回单

⑤ 自动核销与对账
   ↓ 多键匹配（付款单号、发票号、PO号）
   ↓ 完全核销 or 差额按规则分摊
```

**异常分类与闭环时限**：
| 异常类型 | 闭环要求 |
|----------|---------|
| 数量不符 | T+3 天 |
| 单价不符 | T+5 天 |
| 缺票/错票 | T+10 天 |
| 红冲重开 | T+30 天 |

### 2.4 宝尊的具体痛点场景（会议提炼 + 推导）

会议中李志慧明确说的：
> "三个系统之间的断点导致的人为工作……非经营性的采购导致的中控溯源呈现不出来完整信息流"

翻译成具体业务场景：

**场景还原**：一笔GAP门店装修的非经营性采购
```
1. 业务部门在 OA/采购系统 提交采购申请
2. 合同在 合同管理系统 走签署审批
3. 供应商施工完毕，项目部做验收
4. 供应商开发票，财务在 财务系统 做应付挂账
5. 财务发起付款申请

风控要核查：
  - 这笔采购有没有对应的合同？ → 要去合同系统查
  - 合同约定的金额和实际采购金额一致吗？ → 要对两个系统的数据
  - 验收是否完成？验收数量与采购数量是否一致？ → 可能在项目系统或OA里
  - 发票金额与合同、采购单是否匹配？ → 要去财务系统查
  - 是否有重复付款？ → 要查历史付款记录

每一步都要切换系统、手动查数、人工比对。
一天几十笔、上百笔，200人的团队里大量人力消耗在这里。
```

## 三、Agent 可落地的风控场景矩阵（深度版）

> 不再是泛泛的"场景名+一句话描述"，每个场景都要讲清：**业务是什么、现在怎么做、痛在哪、Agent 怎么解决**

### A. 已做 POC 的核心场景

#### 场景 1：非经营性采购跨系统溯源（已做 POC）

| 维度 | 内容 |
|------|------|
| **业务是什么** | 宝尊每月有大量非经营性采购（门店装修、IT设备、市场营销物料、办公用品），需要风控部核查采购全流程的合规性 |
| **现在怎么做** | 风控人员分别登录合同系统查合同、采购系统查采购单、财务系统查付款记录，手动比对三者是否匹配，结果录入Excel汇总表 |
| **痛在哪** | ① 三个系统数据不互通，一笔业务要查三次 ② 人工比对容易遗漏不一致项 ③ 查完无法留痕，审计时还要重新拉数据 |
| **Agent 怎么解决** | Agent 自动对接三个系统的数据源，按业务单号拉取全链路信息（采购单→合同→验收→发票→付款），自动标注断点和不一致项，生成带证据链的核查报告 |
| **对客户的价值话术** | "现在你的人查一笔要20分钟切三个系统，Agent 10秒出结果，还自动标红异常项，审计底稿一键导出" |

### B. 高优先级扩展场景

#### 场景 2：付款前三单匹配自动校验

| 维度 | 内容 |
|------|------|
| **业务是什么** | 每笔付款前必须完成"采购订单-入库验收单-发票"三单匹配，校验数量（差异≤2%）、单价（差异≤1%）、税率（100%一致） |
| **现在怎么做** | 财务人员手动在系统中逐笔核对，超阈值的人工标记并提交主管复核 |
| **痛在哪** | ① 日均付款笔数多，纯人工校验效率低 ② 阈值判断靠经验，标准不统一 ③ 异常处置缺乏标准化SLA |
| **Agent 怎么解决** | Agent 自动抓取三单数据，按预设规则（数量±2%、单价±1%、税率100%）实时匹配，超阈值自动冻结并生成异常工单，按异常类型分配责任人和闭环时限 |

#### 场景 3：四流一致性审查（合同-资金-发票-物流）

| 维度 | 内容 |
|------|------|
| **业务是什么** | 金税四期要求商贸企业四流一致，宝尊作为双上市公司更需确保：合同主体=付款方=发票抬头=物流收发方 |
| **现在怎么做** | 审计前集中突击核查，人工逐笔拉四类单据装订底稿 |
| **痛在哪** | ① 四类单据分散在不同系统 ② 突击核查时间紧张、质量难保证 ③ 一旦发现不一致，追溯成本极高 |
| **Agent 怎么解决** | Agent 持续监控每笔交易的四流数据，实时比对主体一致性，发现异常立即预警，自动生成四合一底稿，日常化替代突击式审查 |

#### 场景 4：合同智能审查

| 维度 | 内容 |
|------|------|
| **业务是什么** | 宝尊代运营450+品牌，自营GAP/HUNTER 341家门店，合同量大（品牌合作协议、供应商合同、租赁合同、劳务合同等） |
| **现在怎么做** | 法务/风控逐份审阅关键条款（付款条件、违约责任、保密条款、竞业限制、知识产权归属） |
| **痛在哪** | ① 合同量大，人工审阅慢 ② 关键条款漏审风险高 ③ 与历史合同对比（条件是否恶化）靠记忆 |
| **Agent 怎么解决** | Agent 提取合同关键条款，与企业标准条款库比对，标记偏离项（付款账期变长、违约金降低等），自动生成审查意见，历史合同条件变化自动高亮对比 |

#### 场景 5：供应商画像与风险预警

| 维度 | 内容 |
|------|------|
| **业务是什么** | 宝尊的供应商涵盖物流商、仓储服务商、营销供应商、门店装修商、IT供应商等，需持续监控资质和信用 |
| **现在怎么做** | 年度集中做供应商准入审查，日常靠采购部门自行把控 |
| **痛在哪** | ① 年度审查只是时点检查，两次审查之间出了问题发现不了 ② 工商变更、行政处罚、经营异常等信息无人持续关注 ③ 关联供应商识别困难 |
| **Agent 怎么解决** | Agent 每日自动抓取企查查/天眼查工商信息，监控所有在库供应商的经营状态变化，出现异常（股权变更、被执行、行政处罚、注销/吊销）立即预警，并自动识别供应商之间的关联关系（同一法人、同一地址、交叉持股） |

#### 场景 6：重复付款检测

| 维度 | 内容 |
|------|------|
| **业务是什么** | 同一笔业务由于系统断点或人为失误，在不同系统发起了重复付款 |
| **现在怎么做** | 月末或审计前对账时才可能发现，发现后追回成本高 |
| **痛在哪** | ① 跨系统数据不互通，同一笔业务在不同系统的编号可能不同 ② 发现滞后，资金已付出 |
| **Agent 怎么解决** | Agent 在付款审批前自动做跨系统去重校验：按供应商+金额+日期范围+业务描述做模糊匹配，疑似重复项冻结并提示人工确认 |

#### 场景 7：大额/异常付款实时预警

| 维度 | 内容 |
|------|------|
| **业务是什么** | 超出正常范围的付款行为（单笔超限、非工作时间付款、同一供应商短期高频付款、付款金额远超合同金额） |
| **现在怎么做** | 依赖审批流程中的人工判断 |
| **痛在哪** | ① 审批人在大量单据中容易审美疲劳 ② 缺乏历史基线比对 ③ 事后审计才发现异常 |
| **Agent 怎么解决** | Agent 建立每个供应商、每个费用类别的历史付款基线，实时比对当前付款申请，偏离基线超阈值（如金额>均值2倍、频次>月均3倍）自动触发预警和升级审批 |

### 4.2 技术方案

```

├─────────────────────────────────┤

```

两个落地场景：

## 五、技术路径：跟技术部对话时要回答的硬问题

### 5.1 "你的 Agent 怎么从我们系统里拿数据？"

```
方案优先级：
1. API 对接（最优）
   → 宝尊系统已微服务化，大概率有 REST API
   → Agent 通过标准 API 调用拉取数据
   → 需要技术部提供 API 文档和权限

2. 数据库直连（次优）
   → 直接查询业务系统数据库的只读副本
   → 需要DBA配合开权限和建视图
   → 注意：不动生产库，只读从库

3. RPA 屏幕操作（兜底）
   → 对于没有 API 也不开放数据库的老旧系统
   → Agent 模拟人工登录系统、查询、提取数据
   → 速度较慢但零侵入
```

### 5.2 "需不需要改造我们现有系统？"

**核心回答：不需要。**

Agent 是"中间层"，不改任何现有系统：
- 合同还在合同系统审批
- 采购还在采购系统走流程
- 付款还在财务系统执行
- Agent 只做**读取 → 对齐 → 分析 → 呈现**

数据结果的呈现方式可灵活选择：
| 方式 | 适用场景 | 改造成本 |
|------|---------|---------|
| **独立前端页面**（当前POC方式） | 快速上线、POC验证 | 零改造 |
| **推送到现有系统**（如飞书/钉钉/OA） | 让用户在熟悉环境中接收信息 | 需简单webhook对接 |
| **嵌入现有系统界面** | 最佳体验但需对方系统支持iframe/插件 | 中等改造 |

**会议中李志慧的提醒**：如果推数据回原系统，对方系统需要承接数据的能力，可能引发改造需求。POC阶段先走独立页面，验证价值后再讨论集成方式。

### 5.3 "你们用什么模型？成本怎么算？"

| 层面 | 答案 |
|------|------|
| 模型 | 优先使用宝尊**已部署**的火山引擎和千问，零额外模型成本 |
| 算力 | 宝尊科技团队从已有资源池分配，不需要额外采购 |
| 我方收费 | POC免费 → 正式阶段按人天计费（3000-3500/人天）→ 按Agent月结 |
| 定价逻辑 | 一个Agent=一个场景，确认需求后评估人天数（如40-50人天），人天×单价=该Agent费用 |
| 框架采购 | 统一价格模型，月结，客户按场景分批投入，风险可控 |
